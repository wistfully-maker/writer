# 设计决策记录

## 核心架构

### 为什么是三个 Agent 而不是六个或七个？
六轮讨论的结论：概念上的角色区分（世界观师、导演、演员、风格师）在实践中边界模糊，
会导致 Agent 之间互相覆盖职责，且增加 LLM 调用成本。
三个核心 Agent（架构师、作家、读者）覆盖了创作流程的三个本质不同的思维模式：
结构思维、创作思维、体验思维。其他角色的功能已合并其中。

### 为什么用 CLAUDE.md + Obsidian 而不是写 Python？
MVP 的核心假设是"多角色协作能否产出更好的文本"，不是"能否搭出多 Agent 框架"。
Claude Code 天然就是编排器+运行时，Obsidian 天然就是 Wiki。
写代码验证假设的成本是几天，用现有工具是几小时。

### 为什么没有编辑 Agent？
MVP 阶段由人类担任编辑角色。
编辑需要做两件事：硬约束校验（已通过 prompt 中的 Rules 从源头预防）
和语言修改（在 MVP 中人类可以自己做或指挥作家做）。
等流程验证通过后，再引入编辑 Agent 降低人类负担。

## 关键设计原则

### 硬约束从源头预防，不是事后检查
不建规则引擎，不搞关键词匹配。
把所有硬约束（人物 Rules）直接注入作家 prompt，让作家在写作时就遵守。
编辑 Agent 作为第二道防线，处理约束太复杂无法在 prompt 中全部列出的情况。

### 读者 Agent 全程贯穿，不是终审
读者反馈不是最后才出现的屏障，而是贯穿每个阶段的校准信号。
大纲阶段评估"这个故事吸引我吗？"，人物阶段评估"我会在乎这个角色吗？"，
每个场景后评估"体验如何？"。

### Prompt 是系统的一切
系统的智能不在代码里，不在架构里，在 prompt 里。
prompt 需要版本管理（frontmatter 中的 prompt_version），
需要迭代优化（基于读者反馈中的高频问题），
需要对抗性测试（新 prompt vs 旧 prompt 盲评）。

### 编译即创作
架构师把 raw/ 编译为 wiki/ 的过程不是格式转换，是第一次创作性加工。
拆解人物三层模型、识别伏笔机会、设计场景分割点——这些都是创意决策。

## 被否决的设计

| 设计 | 否决理由 |
|------|---------|
| 离线推演/在线生成分层引擎 | 过度设计。用架构师"按需探索"替代常驻离线层。 |
| 沉寂伏笔唤醒 Agent | 合并为架构师巡查流程的一个检查步骤。 |
| 多个演员Agent并行写同一场景不同视角再拼接 | 产出"弗兰肯斯坦文本"，不现实。 |
| 硬约束规则引擎（正则匹配） | 中文语义场景下关键词匹配准确率太低。 |
| 灵感Agent自评估可行性 | LLM自我评估不可靠。改为限量生成+架构师评估。 |
| 活性权重知识图谱 | 过度设计。用 last_relevant_chapter 字段+定期清理替代。 |

## 未来扩展路径（验证通过后）

- Phase 2：引入编辑 Agent，减少人类编辑负担
- Phase 3：引入灵感 Agent，注入意外创意
- Phase 4：引入风格 Agent，统一全书语言质感
- Phase 5：渐进式自主权（Level 1-4）
- Phase 6：统计硬指标监控（TTR、句式方差、AI腔指纹）
- Phase 7：成本追踪和模型分派优化
